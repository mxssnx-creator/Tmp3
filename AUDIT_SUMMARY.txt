MAJOR PROBLEMS FOUND
================================================================================

1. CONNECTION MANAGEMENT CHAOS
   ❌ Connections auto-reassign without user action
   ❌ Multiple creation paths cause duplicates
   ❌ State gets corrupted on refresh
   ❌ Deletions don't persist
   Impact: User loses control over which connections process

2. ENGINE PROCESSING RACE CONDITIONS
   ❌ Duplicate timers created (2-3 sets per start)
   ❌ Prehistoric data reloaded unnecessarily (3-5x per minute)
   ❌ Progress counters reset, losing historical data
   ❌ Engine restarts on every page refresh
   Impact: Unpredictable behavior, wasted API calls, lost metrics

3. DASHBOARD TOGGLE IGNORED
   ❌ Turning OFF doesn't stop engine
   ❌ Turning ON doesn't start engine
   ❌ UI shows OFF but processing continues
   Impact: User cannot control engine state via UI

4. CONNECTION COUNT MISMATCHES
   ❌ Dashboard shows wrong numbers
   ❌ Backend state differs from display
   ❌ No single source of truth
   Impact: User confusion about system state

5. DATABASE STRUCTURE FRAGMENTED
   ❌ Keys scattered across 5+ patterns
   ❌ No consolidated progression state
   ❌ Missing indexes (O(n) instead of O(1))
   ❌ No data lifecycle management
   Impact: Performance degradation, memory bloat

6. STARTUP FLOW CHAOS
   ❌ 3 separate engine start entry points
   ❌ Prehistoric loading blocks startup
   ❌ QuickStart auto-enables without consent
   ❌ Settings get overwritten
   Impact: Unpredictable behavior on server restart

ROOT CAUSE ANALYSIS
================================================================================

Problem 1: Five overlapping state fields confuse the system
  ├─ is_inserted (should be clear "exists")
  ├─ is_enabled (should be clear "usable")
  ├─ is_active_inserted (CONFUSING - renamed to "is_assigned")
  ├─ is_enabled_dashboard (should be clear "processing enabled")
  └─ is_predefined (template vs user-created)
  
  → Solution: Rename for clarity + implement true independence

Problem 2: Migrations 016-018 fight each other every startup
  → Solution: Consolidate into single idempotent migration

Problem 3: No engine startup lock prevents duplicates
  → Solution: Add lock set to coordinator

Problem 4: Prehistoric data has no cache check
  → Solution: Cache flag in Redis with TTL

Problem 5: Progress state gets destroyed on restart
  → Solution: Preserve counters, only update timestamp

Problem 6: Multiple progression key patterns exist
  → Solution: Consolidate to single pattern with all data

Problem 7: No indexes for common queries
  → Solution: Add Redis sets for O(1) lookups

Problem 8: Dashboard toggle doesn't trigger engine action
  → Solution: Call stopEngine()/startEngine() in toggle endpoint
